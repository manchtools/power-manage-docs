# power-manage-sdk

## Overview

The SDK defines the contract between all power-manage components. It consists
of three layers:

1. **Protobuf definitions** (`proto/pm/v1/`) — the wire format and service API.
2. **Generated code** (`gen/go/`, `gen/ts/`) — Go and TypeScript stubs produced
   by `buf generate`. Never hand-edited.
3. **Capability library** (Go packages at the module root) — a
   dependency-injected system management library for the agent. Package
   managers, systemd, users/groups, encryption, networking, and more.

## Proto surface

Six proto files define the entire wire API:

| File | Purpose | Key contents |
|------|---------|-------------|
| `common.proto` | Shared identifiers and status enums | `ActionId`, `DeviceId`, `ExecutionStatus`, `AssignmentMode` |
| `actions.proto` | Action types and parameters | 23 action types (Shell, ScriptRun, Package, Service, File, Directory, Reboot, User, Group, SSH, SSHD, AdminPolicy, LUKS, WiFi, Encryption, AgentUpdate, etc.) |
| `agent.proto` | Agent ↔ Gateway bidirectional stream | `AgentService` with `SyncActions`, `ValidateLuksToken`, `RenewCertificate` |
| `control.proto` | Web UI / CLI API | `ControlService` with ~164 RPCs for users, devices, actions, roles, groups, IdPs, SCIM, TOTP, compliance, audit, search |
| `device_auth.proto` | Agent enrollment | `DeviceAuthService` via Unix socket at `/run/pm-agent/enroll.sock` |
| `internal.proto` | Gateway ↔ Control internal API | `InternalService` for credential-bearing operations (LUKS keys, LPS passwords, terminal tokens) |

### Proto conventions

- **Every field crossing a trust boundary carries `@gotags validate:"..."` tag.**
  Required alone is insufficient — type, format, length, and range constraints.
- **IDs are ULIDs** (`string value = 1` with `validate:"required,ulid"`).
- **Enums start at 1** (proto3 requires 0 for unspecified, which is always the
  default/unset value).
- **`oneof` for action parameters** — each action type has its own params message
  inside a oneof, so the wire format carries exactly the right shape.
- **Backward compatibility** — `buf breaking` enforces no wire-breaking changes
  against the main branch. Field numbers are frozen once released.

## Generated code

### Go (`gen/go/pm/v1/`)

Generated by `buf generate` using `protoc-gen-go` and
`protoc-gen-connect-go`. Produces:

- Message structs with `@gotags` validate tags surfaced as struct tags.
- Connect-RPC service clients and handlers (`*connect.Client`,
  `*connect.ServiceHandler`).
- JSON/Proto marshaling via `protojson` and `proto` packages.

### TypeScript (`gen/ts/pm/`)

Generated by `buf generate` using `@bufbuild/protoc-gen-es` and
`@connectrpc/protoc-gen-connect-es`. Produces:

- ES module message classes with `@gotags` embedded in field options.
- Connect-RPC transport-agnostic clients (`createPromiseClient`).
- TypeScript type definitions for all messages and enums.

### Regeneration

```bash
cd sdk && make generate   # regenerates both Go and TypeScript
```

Generated files are committed. CI verifies regeneration produces no diff
(meaning the generated code matches the proto source).

## Capability library (Go)

The SDK provides a dependency-injected system management library organized
by capability. Each capability follows the same shape:

```
Runner (injected) + Backend (explicit) → Manager (stateless handle)
```

### Design principles

1. **Explicit over clever.** The caller always names the privilege tool
   (`exec.Sudo`, `exec.Direct`, `exec.Doas`) and the backend (`pkg.Apt`,
   `service.Systemd`). Behavior never depends on what's installed.
2. **No global state.** Backend selection lives on the instance, not in a
   package variable.
3. **Testable without a host.** The injected `Runner` can be a
   `FakeRunner` that asserts exact argv shapes without real execution.
4. **Mutations return exec.Result.** Every operation returns stdout,
   stderr, and exit code so the caller surfaces what actually happened.

### Capability catalog

| Package | Capability | Backends |
|---------|-----------|----------|

### Testing strategy

- **Unit tests**: `FakeRunner` from `exectest` — asserts exact argv shape,
  scripted stdout/stderr, and error injection. No container, no sudo, no
  network.
- **Integration tests** (`//go:build integration`): Run real tools if
  installed. Skip silently if absent.
- **Container tests** (proposed): Multi-distro Docker stages that test
  against real binaries in known system states (stale locks, degraded
  services, missing tools). See `sdk/test/` design.

## Cryptographic primitives

The `crypto` package provides constant-time comparison, domain-separated
AEAD/HKDF wrappers, and ULID generation. Used by both the server (HMAC
signing of Asynq tasks, secret encryption) and the agent (certificate
key generation).

## Adversarial testing

The `adversary` package contains attack simulations that verify protocol
invariants: forged task signing, key substitution, cross-actor binding
attacks, replay attempts, and downgrade attempts.

## Architecture fitness functions

The `archtest` package contains compile-time and test-time assertions
that structural invariants hold: no circular imports, package layering
rules, proto field validation coverage, and generated-code integrity.

## Invariants

1. **Proto files are the source of truth.** Generated code is never
   hand-edited.
2. **Every proto field crossing a trust boundary has a validate tag.**
3. **Backward compatibility.** `buf breaking` must pass against main.
4. **No global state in the capability library.** All state is injected.
5. **Backend selection is explicit.** Never auto-detect.
6. **All crypto calls carry domain-separation info tags.**
7. **ULIDs for all identifiers.** Never `crypto.randomUUID()`.
8. **docref anchors for all exported symbols** referenced in SDK docs.
